1. Introduction

The Forge Cursor Plugin connects your IDE to Forge (SoftwareForge) for work-order-driven AI-assisted development. It packages skills, slash commands, guard hooks, rules, and a Forge agent so developers can pick tasks, load project context, commit with traceability, create pull requests, and sync activity — all from Cursor Agent chat.

Plugin vs MCP

ComponentWhat it providesHow you get it
Forge PluginSlash commands, skills, hooks, rules, agentTeam Marketplace, public marketplace, or local install
Forge MCP26 API tools (work orders, artifacts, dev activity)Separate one-time setup via Forge UI or ~/.cursor/mcp.json
Important: The plugin does not include MCP credentials. Each developer must connect Forge MCP once. Use /forge-connect or /forge-status to verify.

Prerequisites

  • Cursor 2.6 or later (Agent chat required for slash commands)
  • Forge account at app.softwareforge.ai
  • Forge MCP connected (green under Settings → Tools & MCP, usually user-forge)
  • Plugin enabled under Customize → Plugins → Forge

2. Getting Started

Install the plugin

Team Marketplace (enterprise): Your Cursor admin imports https://github.com/securefactory/forge-cursor-plugin in Dashboard → Settings → Plugins → Team Marketplaces. See the Team Marketplace Setup Guide for admin steps.

Public marketplace: Search for Forge on cursor.com/marketplace once listed (plugin ID: forge).

Local development:

git clone https://github.com/securefactory/forge-cursor-plugin.git
cp -R forge-cursor-plugin/plugins/forge ~/.cursor/plugins/local/forge
# Developer: Reload Window

Enable the plugin

  1. Open Customize in the Cursor sidebar
  2. Find Forge from your team marketplace
  3. Toggle it on (if policy is Default Off)
  4. Run Developer: Reload Window

Slash commands appear in Agent chat only — type /forge to see all eight commands.

Connect Forge MCP

Preferred — one-click from Forge UI:

  1. Log into app.softwareforge.ai
  2. Go to Install in IDE → Cursor from any of:
    • Application Context → Connect IDE tab
    • Project Settings → Connect IDE
    • User Settings → API Tokens
  3. Accept the MCP install prompt in Cursor
  4. Confirm Settings → Tools & MCP shows Forge connected (green)

Manual — user/global ~/.cursor/mcp.json:

{
  "mcpServers": {
    "forge": {
      "type": "streamable-http",
      "url": "https://app.softwareforge.ai/api/mcp",
      "headers": {
        "Authorization": "Bearer forge_..."
      }
    }
  }
}

Create a token under Forge Connect IDE or User Settings → API Tokens. The same JSON shape works in project .cursor/mcp.json for team-scoped MCP.

Verify connection

  1. Open Agent chat
  2. Run /forge-status
  3. Review the readiness report: MCP server, active project, work-order access

First session

After MCP is connected, the agent calls configure_repo once per session before work-order tools:

{ "ide": "cursor", "hooks_already_present": true }

Then select a project with set_project. The easiest path is /forge-connect, which handles both steps.

Quick start

  1. /forge-connect — connect MCP and select project
  2. /work — pick or show your next work order
  3. /start-work — full implement → commit → PR → complete workflow

Troubleshooting

Forge Agent — purpose and usage

Purpose: AI development workflow assistant that manages work orders, tracks activity, and maintains project context from within the IDE.

Usage: Invoke when you need structured help with task selection, implementation scoped to a work order, formatted commits ([WO-...]), PR creation, or project setup. The agent follows the 12-step end-to-end lifecycle and applies forge-conventions rules automatically.

Capabilities: All 26 MCP tools across work orders (15), dev activity (3), and project context (9).

Rules — purpose and usage

Purpose: Enforce Forge workflow conventions so every agent session follows the same discipline — session gates, artifact loading, linked-repo commits, RTM drift checks, and activity reporting.

Usage: Always active in the background. You do not invoke rules manually; they guide agent behavior automatically. Key conventions:

  • configure_repo once per session before work-order tools
  • set_project before work-order or activity operations
  • Load artifacts before implementing scoped work
  • Only commit/PR to list_linked_repos
  • prepare_commit before committing; update_work_order after each meaningful commit
  • sync_dev_activity after push/PR and before complete_work_order
  • No scope changes outside the active work order without explicit approval

5. Slash Commands

All commands run in Agent chat. Type /forge to browse. Nearly all commands require a connected Forge MCP server — on failure they direct you to /forge-connect.

Command picker

If you need…Use
MCP not connected/forge-connect
Diagnose connection/forge-status
Project briefing/forge-context
One document or keyword search/forge-artifacts
Browse backlog / stats/forge-orders
Next task (lightweight)/work
Full implementation workflow/start-work
Sync git activity to Forge/forge-sync

/forge-connect

Purpose: Install/connect Forge MCP, verify auth, configure session, and select active project.

When to use: MCP missing, auth fails, no project set, or any command hits the MCP prerequisite gate.

Usage: Run once per developer machine after install, or whenever /forge-status reports disconnected. Does not implement code — only sets up the session.

Steps:

  1. Discover Forge MCP servers from user and project config
  2. Test with list_my_projects
  3. If fail: guide through Forge Install in IDE → Cursor
  4. Call configure_repo with hooks_already_present: true
  5. list_my_projects → user picks → set_project
  6. Confirm via list_linked_repos

MCP tools: list_my_projects, configure_repo, set_project, list_linked_repos

/forge-status

Purpose: Validate live Forge MCP connection and print a readiness report. Read-only — does not configure anything.

When to use: First-time setup check, troubleshooting, or before relying on other Forge commands.

Usage: Safe to run anytime. Use the report to see which MCP server is active, whether work-order tools respond, and which project is selected.

Steps:

  1. Discover Forge MCP servers from ~/.cursor/mcp.json and .cursor/mcp.json
  2. Call list_my_projects on each candidate until one succeeds
  3. Test work-order access via get_work_order_stats or list_work_orders
  4. Check active project; print status report

MCP tools: list_my_projects, get_work_order_stats, list_work_orders, get_project_state

/forge-context

Purpose: Load project context before implementing work — journey state, linked repos, and core artifacts.

When to use: You need a briefing on intent, PRD, architecture, and RTM without picking a work order yet.

Usage: Run at the start of a feature or when joining a project. Produces a concise briefing the agent uses for subsequent implementation.

Steps:

  1. Confirm MCP connection, then configure_repo → set_project
  2. get_project_state
  3. list_linked_repos
  4. get_artifact for intent, prd, architecture, rtm
  5. Optional search_artifacts for topic queries

MCP tools: get_project_state, list_linked_repos, get_artifact, search_artifacts, list_ux_references

/forge-artifacts

Purpose: Fetch or search Forge journey artifacts by type or keyword.

When to use: You want a specific document (PRD, BRD, RTM) or keyword search without the full briefing from /forge-context.

Usage: Ask for a document type or search term; agent calls get_artifact or search_artifacts and summarizes unless you request the full document.

Artifact types: intent, prd, brd, architecture, work_orders, rtm

MCP tools: get_artifact, search_artifacts, list_ux_references

/forge-orders

Purpose: Browse and inspect work orders without starting full implementation.

When to use: Stats, filtered lists, task details, comments, or clarifications — read-only backlog exploration.

Usage: Explore the backlog, check team comments, or post clarifications without entering the implement → commit pipeline.

Steps:

  1. get_work_order_stats and get_workflow_stages
  2. list_work_orders with optional status/priority filters
  3. get_work_order for a specific task
  4. Optional: get_work_order_comments, comment_on_work_order, get_clarifications, ask_question

MCP tools: get_work_order_stats, get_workflow_stages, list_work_orders, get_work_order, comment and clarification tools

/work

Purpose: Show the current work order or pick the next one — quick "what should I work on?"

When to use: Lighter than /start-work — claim or view a task without the full implement → PR pipeline.

Usage: Daily standup-style check-in. Agent picks or displays the next WO and offers to hand off to /start-work when you are ready to code.

Steps:

  1. get_next_work_order (auto-transitions) or browse via list_work_orders
  2. get_work_order for details and RTM traceability
  3. Optional artifact context; offer /start-work to begin implementation

MCP tools: get_next_work_order, get_work_order, get_workflow_stages, list_work_orders, transition_work_order

/start-work

Purpose: Full end-to-end workflow from task selection to PR creation and completion.

When to use: Ready to implement a work order through to merged PR and Forge status update.

Usage: Primary command for feature development. Agent loads context, helps implement, enforces RTM drift check, commit discipline, PR linking, activity sync, and completion.

Steps:

  1. configure_repo → set_project
  2. get_next_work_order + get_work_order
  3. Load context: get_project_state, artifacts, search_artifacts, list_ux_references
  4. list_linked_repos — clone before coding
  5. Implement and test
  6. RTM drift check → rtm_drift_summary
  7. prepare_commit → commit with [WO-...] message
  8. update_work_order with dev-activity fields
  9. Push → create_pull_request → run returned CLI
  10. sync_dev_activity
  11. complete_work_order after PR merges

MCP tools: All project-context, work-order, and dev-activity tools (see Workflows section)

/forge-sync

Purpose: Sync commits and PRs from linked repos into Forge.

When to use: After git push or PR creation; before complete_work_order; when Forge activity looks stale.

Usage: Run after every push or when the pre-push hook reminds you. Validates and replays gaps if data is incomplete.

Steps:

  1. sync_dev_activity (optional repo_name, mode)
  2. If stale: validate_dev_activity_sync
  3. If gaps: replay_dev_activity

MCP tools: sync_dev_activity, validate_dev_activity_sync, replay_dev_activity

Workflow conventions

  • Call configure_repo once per session before work-order tools
  • Call set_project before work-order or activity operations
  • Load artifacts before implementing scoped work
  • Only commit/PR against list_linked_repos
  • Call get_workflow_stages before manual transitions or status filters
  • Use prepare_commit before committing
  • Run RTM drift check before Forge-tagged commits; include rtm_drift_summary in update_work_order
  • Call update_work_order after each meaningful commit
  • Link PRs via create_pull_request
  • Sync with sync_dev_activity after push/PR and before complete_work_order
  • Do not change scope outside the active work order without explicit approval

End-to-end work order lifecycle

  1. configure_repo — session hooks gate
  2. set_project — select active project
  3. get_next_work_order — pick highest-priority backlog task
  4. Load context: get_work_order + get_artifact (intent, prd, architecture, rtm)
  5. list_linked_repos — clone repos
  6. Implement and test (scoped to work order)
  7. RTM drift check → prepare rtm_drift_summary
  8. prepare_commit → commit with [WO-...] message
  9. update_work_order — full dev-activity fields
  10. Push → create_pull_request → run CLI command
  11. sync_dev_activity
  12. complete_work_order — after PR merges and syncs

Comments vs clarifications

Team Q&Aask_question, get_clarificationswork_order_id (UUID) or artifact_id
Most other WO toolsget_work_order, update_work_order, etc.work_order_id (UUID)

Requirements traceability check (RTM)

Before Forge-tagged commits, confirm your implementation still matches the requirements traceability matrix (RTM):

  1. Call get_artifact with type rtm
  2. Find RTM rows linked to the current work order (also in get_work_order → traceability)
  3. Compare implementation against PRD and architecture
  4. Record in update_work_order field rtm_drift_summary

7. Safety Hooks

Safety hooks run automatically when the Forge plugin is enabled. View them under Settings → Hooks.

HookEventBlocking?Behavior
Pre-commit git commit Yes Applies only to commits containing [WO-] or [TASK-]. Blocks until checklist complete: (1) run/affect unit tests, (2) validate acceptance criteria via get_work_order, (3) report via update_work_order, (4) create .forge-commit-ready marker file, then retry. Defers to .cursor/hooks/forge-pre-commit.sh if present (installed by MCP configure_repo without plugin).
Pre-push git push No (advisory) Always allows push; reminds agent to run sync_dev_activity and link PRs via create_pull_request.
Destructive command guard Shell commands Yes / Ask Denies: git push --force, rm -rf /. Asks for approval: kubectl apply to prod. All other commands allowed.
Sensitive file warning File reads No Warns on filename patterns: *.env, *credentials*, *secrets*, *.pem, *.key, SSH key names. Does not scan file content.
Session complete Agent stop No One-time reminder to sync_dev_activity when session completes successfully. Uses a reminder flag file so the message is not repeated.

Hook integration with MCP

When the marketplace plugin is installed, call configure_repo with:

{ "ide": "cursor", "hooks_already_present": true }

This tells Forge that plugin hooks are active and avoids duplicate project-level hook installation.

Requirements

  • Three hooks require Python 3 for JSON parsing
  • All hook scripts use bash
  • Hooks resolve paths via ${CURSOR_PLUGIN_ROOT}
